## 2. THE GEOMETRIC FAULT LINES: THE "QUAD-SQUEEZE"
The K2 protocol fails to maintain a 1:1 balance between user deposits and protocol debt due to the hardcoded misuse of `ray_div_down` (floor rounding) and `ray_div_up` (ceiling rounding) across four primary transaction nodes.

**Fault A: The Entry Squeeze (Minting A-Tokens)**
When a user supplies underlying capital to the protocol, the system routes the transaction through `ray_div_down`. By truncating the division, the system deliberately mints the absolute minimum number of pool shares for the user. The fractional remainder is absorbed by the protocol.

**Fault B: The Exit Squeeze (Burning A-Tokens)**
When a user withdraws their capital, the system reverses its logic and routes the transaction through `ray_div_up`. This ceiling calculation forces the user to burn the absolute maximum number of shares to retrieve their original supply, guaranteeing a loss of fractional equity.

**Fault C: The Debt Squeeze (Borrowing)**
When a user opens a loan, the protocol calculates the debt issuance using `ray_div_up`. The ceiling bias instantly inflates the user's debt obligation from the very first frame of the transaction.

**Fault D: Phantom Debt (Repayment)**
When a user repays their loan, the protocol routes the transaction through `ray_div_down`. By truncating the repayment power, the protocol shaves off less debt than the user actually paid for. This specific fracture leaves behind a permanent fractional residue known as "Phantom Debt."

## 3. ROOT CAUSE ANALYSIS (`utils.rs`)
The core of the vulnerability lives entirely within the `contracts/shared/src/utils.rs` mathematical layer. 

* **Truncation Bias:** In `ray_div_down`, the protocol executes a raw integer division. In this environment, this blindly chops off the remainder.
* **Ceiling Bias:** In `ray_div_up`, the protocol adds the denominator minus one before dividing. This artificially forces the integer to the next whole number, ensuring the mathematical advantage always favors the protocol over the user.

*Note: The presence of a manual developer patch (commented as L-09 in the source code) designed to blindly delete debt balances of 1 unit or less indicates an awareness that the protocol's base mathematics do not naturally balance to zero.*
